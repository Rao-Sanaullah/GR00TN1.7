GR00T N1.7 on WSL2 Ubuntu 22.04

A battle-tested installation guide for NVIDIA Isaac GR00T N1.7 fine-tuning on Windows Subsystem for Linux with an RTX 5090. Includes real fixes for corporate SSL, permission errors, and the Blackwell Triton patch.

Before You Begin
⚠ RTX 5090 (Blackwell / sm_120) — Triton Patch Required Triton 3.3.1 pinned by PyTorch 2.7 does NOT support sm_120. Fine-tuning will crash with RuntimeError in ptx_get_version() without Step 8.
Corporate Proxy / VPN? If your machine sits behind a corporate firewall (Zscaler, etc.), you'll hit SSL UnknownIssuer errors. Steps 5 covers the CA cert fix.
CUDA Drivers on Windows WSL2 shares the Windows GPU driver automatically. Do NOT install a separate CUDA driver inside WSL — only the CUDA toolkit if nvcc is missing.
Installation Steps
01
Verify CUDA & GPU Inside WSL2 nvidia-smi · nvcc · python3.10
▶

Open your WSL2 terminal and confirm the GPU and CUDA toolkit are visible:

$bash
# Must show RTX 5090 + driver version (e.g. 570.xx)
nvidia-smi

# Should show: release 12.8
nvcc --version

# Confirm Python 3.10
python3.10 --version
nvcc not found? Install the CUDA toolkit: sudo apt install -y nvidia-cuda-toolkit
Or use NVIDIA's official installer at developer.nvidia.com/cuda-downloads (select Linux → x86_64 → Ubuntu → 22.04 → WSL-Ubuntu).
python3.10 not found? sudo apt install -y python3.10 python3.10-venv python3.10-dev
02
Install System Dependencies git-lfs · ffmpeg · build-essential
▶
Install git-lfs BEFORE cloning. Skipping this causes parquet files in demo_data/ to arrive as corrupted LFS pointer stubs — not actual data.
$bash
sudo apt-get update
sudo apt-get install -y git git-lfs ffmpeg curl build-essential

# Activate LFS hooks globally
git lfs install
03
Clone the Repository --recurse-submodules
▶

GR00T uses git submodules for external dependencies. The --recurse-submodules flag fetches everything in one shot:

$bash
git clone --recurse-submodules https://github.com/NVIDIA/Isaac-GR00T
cd Isaac-GR00T
Already cloned without submodules? git submodule update --init --recursive
04
Fix Directory Permissions ⚠ Real issue — root-owned clone
▶
If you cloned with sudo (or as root), the .venv creation will fail: error: failed to create directory .venv: Permission denied (os error 13)

Check ownership first:

$bash
ls -la ~/workspace/Grootn17/

If the directory is owned by root root instead of your user, fix it:

$bash
# Replace ~/workspace/Grootn17/Isaac-GR00T with your actual path
sudo chown -R $USER:$USER ~/workspace/Grootn17/Isaac-GR00T
actual output
$ ls -la ~/workspace/Grootn17/
drwxr-xr-x 3 root root 4096 Apr 22 09:52 Isaac-GR00T
$ sudo chown -R $USER:$USER ~/workspace/Grootn17/Isaac-GR00T
# No output = success
05
Install Corporate CA Certificate ⚠ Real issue — SSL UnknownIssuer behind VPN/proxy
▶
Behind a corporate firewall? You'll see this error: invalid peer certificate: UnknownIssuer
This happens because the company proxy (e.g. Zscaler) intercepts HTTPS and re-signs with its own CA that WSL doesn't trust.

Copy your company's CA certificate (the .cer file) into the WSL trust store. The file must be renamed to .crt in the destination:

$bash
# Adjust path to wherever your CA.cer lives
sudo cp ~/workspace/CA.cer /usr/local/share/ca-certificates/company-CA.crt

# Rebuild the trust store (should print "1 added")
sudo update-ca-certificates
update-ca-certificates output you want to see: 1 added, 0 removed; done. — warnings about "duplicate" entries are harmless.

Then tell uv (and pip/requests) to use the system cert bundle by adding these to your ~/.bashrc:

$bash — persist to ~/.bashrc
echo 'export SSL_CERT_FILE=/etc/ssl/certs/ca-certificates.crt' >> ~/.bashrc
echo 'export REQUESTS_CA_BUNDLE=/etc/ssl/certs/ca-certificates.crt' >> ~/.bashrc
source ~/.bashrc
Why export for each new shell? uv has its own TLS stack (rustls) and reads SSL_CERT_FILE at runtime. Without the export, it ignores the system trust store even after update-ca-certificates.
actual output
$ sudo update-ca-certificates
Updating certificates in /etc/ssl/certs...
rehash: warning: skipping ca-certificates.crt,it does not contain exactly one certificate or CRL
rehash: warning: skipping duplicate certificate in company-CA.pem
1 added, 0 removed; done.
06
Install uv and Sync Environment 137 packages · ~19 min on first run
▶

GR00T uses uv for fast, reproducible dependency management. Install it, then sync the full environment with Python 3.10:

$bash — from inside Isaac-GR00T/
# Install uv
curl -LsSf https://astral.sh/uv/install.sh | sh
source $HOME/.local/bin/env

# Sync (SSL env vars must be set from Step 5 first!)
uv sync --python 3.10
What gets installed: torch 2.7.1+cu128, flash-attn 2.7.4, TensorRT 10.15, deepspeed, transformers, torchcodec, and 130+ more packages. The first run takes ~19 minutes — subsequent runs resolve in seconds.
flash-attn re-validation on every uv run? Normal. uv re-checks the cached wheel URL each time (~2–3s). It is NOT rebuilding from source.
actual output (truncated)

07
Set CUDA_HOME Required for fine-tuning · deepspeed compilation
▶
Fine-tuning will fail immediately with: CUDA_HOME is unset — deepspeed needs this to compile CUDA extensions.
$bash
export CUDA_HOME=/usr/local/cuda
echo 'export CUDA_HOME=/usr/local/cuda' >> ~/.bashrc
source ~/.bashrc
Alternatively, run the provided script which sets this and other GPU paths: uv run bash scripts/deployment/dgpu/install_deps.sh
08
Patch Triton for RTX 5090 (Blackwell) ⚠ Critical — sm_120 not supported by Triton 3.3.1
▶
Without this patch, any torch.compile path crashes with: RuntimeError in Triton's ptx_get_version()
Triton 3.3.1 (pinned by PyTorch 2.7) does not recognise CUDA Blackwell architecture sm_120.
$bash
uv run bash scripts/patch_triton_cuda13.sh
actual output
$ uv run bash scripts/patch_triton_cuda13.sh
Uninstalled 1 package in 2ms
Installed 1 package in 8ms
Patched .../triton/backends/nvidia/compiler.py to support CUDA 13.x
Installed triton_cuda13_patch.pth (runtime monkey-patch, survives uv reinstalls)
Why "cuda13"? The script is named for Thor/Spark CUDA 13.x platforms but fixes the same sm_ version detection bug that hits sm_120 on CUDA 12.8. Run it regardless of your CUDA version.
Still hitting torch.compile errors after the patch? Add TORCH_COMPILE=0 to your environment or pass --no-compile to the fine-tune script. TensorRT inference is fully supported on Blackwell regardless.

Optionally run a zero-shot inference test on the included demo dataset (downloads the base model ~6GB from HuggingFace on first run):

$bash
uv run python scripts/deployment/standalone_inference_script.py \
    --model-path      nvidia/GR00T-N1.7-3B \

HuggingFace download fails behind proxy? Set: export HF_ENDPOINT=https://huggingface.co and ensure your SSL env vars from Step 5 are still active.
10
Run Fine-tuning on Your Robot Data NEW_EMBODIMENT · single GPU
▶

Replace the dataset path and modality config with your own. The demo below uses the included SO100 5-episode dataset as a smoke test:

$bash — single GPU (RTX 5090)
CUDA_VISIBLE_DEVICES=0 uv run python \
    gr00t/experiment/launch_finetune.py \
    --base-model-path      nvidia/GR00T-N1.7-3B \
    --dataset-path         demo_data/cube_to_bowl_5 \
    --embodiment-tag       NEW_EMBODIMENT \
    --modality-config-path examples/SO100/so100_config.py \
    --num-gpus             1 \
    --output-dir           /tmp/test_finetune \
    --max-steps            2000 \
    --global-batch-size    32 \
    --dataloader-num-workers 4
OOM on 32GB VRAM? The repo recommends 40GB+. Start with --global-batch-size 16 or 8. Smaller batches work fine with proportionally more steps.
Using your own robot data? Replace demo_data/cube_to_bowl_5 with your dataset path and examples/SO100/so100_config.py with your modality config. See getting_started/finetune_new_embodiment.md for the full data format spec.

After training, validate with open-loop evaluation:

$bash
uv run python gr00t/eval/open_loop_eval.py \
    --dataset-path   <YOUR_DATASET> \
    --embodiment-tag NEW_EMBODIMENT \
    --model-path     /tmp/test_finetune \
    --traj-ids 0 1 \
    --action-horizon 16
Output saved to /tmp/open_loop_eval/traj_0.jpeg — ground truth vs predicted actions with MSE. Expect 5–6% variance between runs due to non-deterministic augmentations.

Fine-tuning Tips for RTX 5090
→Start with --global-batch-size 16 — the 32GB 5090 is just under the 40GB+ recommendation; batch-16 keeps VRAM safe
→Train 2,000–5,000 steps for new embodiments; monitor open-loop eval MSE to judge convergence
→--state_dropout_prob defaults to 0.2 in the CLI; lower it if your task depends heavily on proprioceptive state
→N1.7 uses a relative EEF action space — configure your robot actions as deltas from current pose for best cross-embodiment transfer
→Add --use-wandb to the fine-tune command to enable Weights & Biases logging
→If torch.compile still fails post-patch, set TORCH_COMPILE=0 in your environment before running fine-tuning
